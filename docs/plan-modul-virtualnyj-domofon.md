# План: модуль виртуального домофона (vdom) — черновик для следующей итерации

Состояние репозитория откатано к коммиту **до** внедрения vdom. Ниже — что потребуется сделать заново, с учётом прошлых находок.

## 1. Данные и модель устройства

- SQL-миграция: таблица виртуальных домофонов (связь с домом/домофоном, токены панели, метаданные).
- `server/hw/ip/domophone/models/virtual.json` + класс `virtual.php` в том же дереве, как у остальных моделей.

## 2. Сервис и API (PHP)

- Сервис `VirtualDomophoneService` (или аналог): CRUD, выдача `panelUrl`/QR-данных, регенерация токена, тестовое открытие.
- Эндпоинты под `server/api/vdom/` (имена уточнить по конвенции проекта): список, один объект, PUT/POST/DELETE, `regenerateToken`, `testOpen`.
- **Гостевой** `guestManifest` (если нужен публичный доступ без Bearer): в `frontend.php` явно обработать ветку без `$auth`, не использовать `$auth["uid"]` в кэше Redis при `$auth === false`; кэш `CACHE:FRONT:*` только при реальном uid.

## 3. Права и матрица

- Записи в `internal`/`authorization` для методов vdom (как в прошлой попытке).
- Проверка `AVAIL` на клиенте и согласованность с бэкендом.

## 4. Клиентский модуль

- `client/modules/vdom/`: `vdom.js`, `i18n/ru.json`, `en.json`, `version`.
- Пункт меню, таблица/редактор, вызовы `QUERY`/`POST`/`DELETE` с разбором ответа `{ "200": { ... } }` где нужно.
- Статическая страница панели (например `client/vdom/panel.html`), если остаётся требование QR для гостя.

## 5. Конфигурация

- В `config.example.json` и в локальном `config.json`: добавить `"vdom"` в `modules` **после** проверки зависимостей.
- Не ломать порядок модулей: модуль **cs** подписывается на `modules.mqtt.subscribe` — в конфиге должен быть `**mqtt` до `cs`**, либо в `cs.js` оставлять подписки только при `modules.mqtt` (защита без MQTT).

## 6. Ответы API и клиент

- Успешные ответы часто в виде `{ "200": { "ключ": данные } }` (`api::SUCCESS`). В `whoAmI` и аналогичных местах при необходимости нормализовать тело до ожидаемой формы **до** использования полей.

## 7. Проверки перед релизом

- Логин под пользователем с правами, без прав, гость (если есть).
- Загрузка всех модулей из `config.modules` без ошибок в консоли.
- WebSocket `101` — норма, не ошибка.

## 8. Опционально позже

- Временная диагностика (`boot-trace`) — только при отладке; не включать в прод по умолчанию.
- Service worker: при смене логики учитывать `skipWaiting` / кэш и загрузку скриптов с `?ver=`.
