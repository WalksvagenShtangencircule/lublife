# Виртуальная панель: DTMF → HTTP (шлагбаум / калитка)

Чеклист для развёртывания на **новом сервере** (и для агента по памяти). Детали Asterisk — в [10.asterisk.md §6](10.asterisk.md).

## Зачем

- **Кнопка «Открыть»** в мобильном приложении: `POST /mobile/address/openDoor` → сервер делает **GET** по URL из `doorOpeningUrls` панели.
- **DTMF во время разговора** с виртуальной панелью (`model`: `virtual.json`): Asterisk AMI → `server/cli/vdom_ami_dtmf_listener.php` → `POST …/asterisk/extensions/vdomDtmfDoor` → тот же **GET** по URL (цифра = основной/альтернативный DTMF панели).

## Файлы в репозитории

| Путь | Назначение |
|------|------------|
| `asterisk/extensions.lua` | Перед `Dial` с панели: `redis:setex("vdom_call_ctx:"..linkedid, …)` |
| `asterisk/manager.conf` | Секция `[rbtdom]` — логин AMI (секрет = `vdom_dtmf.ami.secret` в `config.json`) |
| `server/asterisk.php` | `case "vdomDtmfDoor"` — валидация + GET по `doorOpeningUrls` |
| `server/cli/vdom_ami_dtmf_listener.php` | Процесс: AMI → Redis контекст → HTTP к `vdomDtmfDoor` |
| `install/systemd/rbt-vdom-ami-dtmf.service` | systemd unit |
| `server/config/config.sample.json5` | Шаблон блока `vdom_dtmf` |

`server/config/config.json` в git **не** хранится (`.gitignore`); после `cp …/config.sample.json5` нужно вручную/скриптом завести `vdom_dtmf`.

## Синхронизация секретов (обязательно)

1. В **`server/config/config.json`** (после `strip-config` из json5):
   - `vdom_dtmf.open_api_key` — длинная случайная строка.
   - `vdom_dtmf.ami.username` = **`rbtdom`** (как в `manager.conf`).
   - `vdom_dtmf.ami.secret` = **точно** тот же пароль, что в **`asterisk/manager.conf` → `[rbtdom]` → `secret`**.

2. В **`asterisk/manager.conf`** для `[rbtdom]`:
   - Достаточно **`permit = 127.0.0.1/255.255.255.255`** при необходимости ограничить localhost.
   - **Не** добавлять строку **`deny = 0.0.0.0/0.0.0.0`** сразу после `permit` — на Asterisk 20 AMI-логин с 127.0.0.1 в таком виде **ломается**.

## Команды после правки конфигов

```bash
sudo asterisk -rx "manager reload"
sudo asterisk -rx "module reload pbx_lua"
```

## systemd (слушатель AMI)

```bash
sudo mkdir -p /var/log/rbt_vdom_ami_dtmf && sudo chown rbt:rbt /var/log/rbt_vdom_ami_dtmf
sudo ln -sf /opt/rbt/install/systemd/rbt-vdom-ami-dtmf.service /etc/systemd/system/rbt-vdom-ami-dtmf.service
sudo systemctl daemon-reload
sudo systemctl enable rbt-vdom-ami-dtmf --now
sudo systemctl status rbt-vdom-ami-dtmf --no-pager
```

Логи процесса: `/var/log/rbt_vdom_ami_dtmf/stderr.log`.  
Успешные срабатывания открытия по DTMF: `server/logs/vdom_dtmf_door.log` (рядом с PHP).

## Диагностика: DTMF есть, открытия нет

1. `journalctl -u rbt-vdom-ami-dtmf -f` или `tail -f /var/log/rbt_vdom_ami_dtmf/stderr.log` — строки **`DTMF matched`** (нашли Redis) или **`DTMF unmatched`** (событие AMI есть, ключ `vdom_call_ctx:*` не подошёл).
2. В `config.json` можно выставить `"vdom_dtmf": { "log_unmatched_dtmf": false }` чтобы отключить шум после починки.
3. После правки `extensions.lua`: `sudo asterisk -rx "module reload pbx_lua"`.

## Быстрая проверка API (без телефона)

Подставьте реальные `domophoneId`, `flatId`, секреты из `config.json`:

```bash
curl -sS -X POST 'http://127.0.0.1/asterisk/extensions/vdomDtmfDoor' \
  -H 'Content-Type: application/json' \
  -d '{"auth":"ВАШ_open_api_key","domophoneId":1,"flatId":1,"digit":"1"}'
```

Ожидается JSON с `"ok":true` или понятной ошибкой (`notFound`, `noDoorForDigit`, …).

## Проверка AMI-логина с хоста

```bash
( printf 'Action: Login\r\nUsername: rbtdom\r\nSecret:ВАШ_СЕКРЕТ_ИЗ_config\r\n\r\n'; sleep 1 ) | nc -q 2 127.0.0.1 5038 | head -5
```

Должно быть `Response: Success`.
