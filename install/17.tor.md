# Tor (SOCKS5) for server diagnostics Telegram

When `api.telegram.org` is blocked, RBT can send diagnostics alerts and use **«Ждать чат»** via a local **Tor SOCKS** proxy (`diagnostics.telegram.tor` in `config.json` — see `server/config/config.sample.json5`).

## 1. Install (Ubuntu / Debian)

```bash
sudo apt-get update
sudo DEBIAN_FRONTEND=noninteractive apt-get install -y tor
sudo systemctl enable tor
sudo systemctl start tor
```

On Ubuntu 24.04+, `tor.service` pulls up **`tor@default.service`** (the SOCKS daemon). If `systemctl enable tor@default` warns about template units, enabling **`tor`** is enough.

## 1c. Быстрый путь для RBT (Ubuntu, Telegram через Tor)

Если Tor «застревает» на bootstrap — сначала мосты, затем права для php-fpm, затем при необходимости TCP Control:

```bash
cd /opt/rbt   # или путь к клону репозитория
sudo bash install/scripts/tor_enable_snowflake.sh    # по желанию, если без мостов Tor не выходит в сеть
sudo bash install/scripts/tor_php_debian_tor.sh      # www-data + debian-tor, перезапуск php-fpm
# опционально: sudo bash install/scripts/tor_append_control_port.sh
sudo systemctl restart tor@default 2>/dev/null || sudo systemctl restart tor
```

Проверка: `curl --socks5-hostname 127.0.0.1:9050 https://check.torproject.org/api/ip --max-time 120`.

## 1b. Firewall и порты (Ubuntu, без ограничений у хостера)

Ниже — что **должно быть доступно** с точки зрения сети. Если что-то из исходящего заблокировано у провайдера/в security-group, Tor «застрянет» на bootstrap (~10–15 %) или curl через SOCKS будет таймаутиться.

### Локально (loopback), не из интернета

| Адрес | Назначение |
|--------|------------|
| **127.0.0.1:9050** | SOCKS5 Tor — сюда подключается PHP (RBT). Слушает только Tor на localhost. **В интернете и на внешнем интерфейсе сервера открывать не нужно.** |
| **127.0.0.1:9051** | TCP ControlPort (если добавлен в `torrc`, см. `tor_append_control_port.sh`) — NEWNYM. Только localhost. **Наружу не открывать.** |
| **`/run/tor/control`** (Unix socket) | Управление Tor на Debian/Ubuntu по умолчанию (альтернатива TCP 9051). Доступ у группы **debian-tor**; для php-fpm: `sudo bash install/scripts/tor_php_debian_tor.sh`. |

Между **nginx/php-fpm** и **Tor** на той же машине дополнительных правил фаервола обычно не требуется (всё через loopback).

### Входящие с интернета

Для **клиента Tor** (не relay, не bridge-сервер) **отдельные входящие порты под Tor не нужны**: соединения с релеями и каталогами инициируются **исходящими** с сервера.

HTTPS и SSH к вашему серверу — по вашей политике; к работе Tor-клиента это не привязано.

### Исходящие (обязательно для работы Tor)

Нужен **нормальный исходящий доступ в интернет** (не только «веб на 443»):

- **Обычный Tor:** исходящие **TCP** к публичным узлам сети (порты у релеев разные — часто **443**, **9001**, и др.; **одного фиксированного порта нет**). Политика «разрешить исходящий TCP почти везде» (как у типичного сервера без жёсткого egress) подходит.
- **Мосты (Snowflake, obfs4 и т.д.):** исходящие **TCP** к адресам и портам из строк `Bridge` в `torrc` (у Snowflake часто задействуется **443** и см. [документацию Tor Project](https://community.torproject.org/relay/setup/bridge/debian-ubuntu/) по транспорту).
- **DNS:** резолв имён для системы/Tor (обычно **UDP/53** или **TCP/53** к резолверу — по умолчанию не режут).

Если у хостера или в **ufw** разрешены **только** исходящие HTTP/HTTPS к «обычным» сайтам, а произвольные исходящие TCP к IP:port релеев Tor — **нет**, то Tor **не построит цепочку**; нужны либо ослабление egress, либо мосты (раздел 2).

### ufw (если включён)

По умолчанию `ufw` разрешает **исходящее** (`default allow outgoing`) — Tor обычно работает. Если включён запрет исходящего по умолчанию, явно разрешите исходящий трафик, достаточный для Tor (и при использовании Snowflake — см. актуальные требования транспорта в документации Tor).

## 2. Check SOCKS

Default SOCKS listen: **`127.0.0.1:9050`**.

```bash
ss -tlnp | grep 9050
curl -sS --socks5-hostname 127.0.0.1:9050 https://check.torproject.org/api/ip --max-time 60
# TLS + имя api.telegram.org через Tor (резолв внутри SOCKS, как rdns=True в aiohttp-socks):
curl -sS --socks5-hostname 127.0.0.1:9050 -o /dev/null -w "HTTP %{http_code}\n" "https://api.telegram.org/" --max-time 90
```

If the second command times out, Tor may be blocked (no relays reachable). Use [bridges](https://community.torproject.org/relay/setup/bridge/debian-ubuntu/) (e.g. **Snowflake** / `obfs4`) — add `UseBridges`, `ClientTransportPlugin`, and `Bridge` lines to `/etc/tor/torrc`, then `sudo systemctl restart tor@default`.

## 3. Webhook и long poll

Если у бота в [@BotFather](https://t.me/BotFather) задан **webhook** (URL), Telegram **не** отдаёт апдейты в `getUpdates` — они уходят на тот URL. Тогда `/start` в чате с ботом **не появится** в RBT, пока webhook не снят (`deleteWebhook`). В UI «Ждать чат» сначала выполняется снятие webhook; **после успеха** отправьте `/start` снова.

## 3b. RBT configuration

1. Copy or merge from `server/config/config.sample.json5` the `diagnostics.telegram` block.
2. Set `enabled`, `bot_token`, `chats` (and optionally `tor.socks_host` / `tor.socks_port` if not default).
3. With strict blocking and **no** usable direct path, set `tor.fallback_direct` to `false` and keep `delete_webhook_direct_first` `false` (defaults in sample).

PHP (php-fpm) connects to `127.0.0.1:9050` on the **same host** as the app — no extra firewall rule for outbound SOCKS to localhost.

## 4. Tor Control (NEWNYM) — рекомендуется для Telegram

Часть выходных узлов Tor **блокирует** `api.telegram.org`. RBT при таймауте/обрыве через SOCKS делает **один** повтор после **`SIGNAL NEWNYM`** (новая цепочка, другой exit), если в `config.json` не отключено `diagnostics.telegram.tor.retry_newnym_on_transport_fail` (по умолчанию включено).

**Debian/Ubuntu:** пакет `tor` обычно слушает **Unix control socket** `/run/tor/control` (не обязательно TCP `127.0.0.1:9051`). RBT подключается к сокету автоматически, если он есть, и аутентифицируется по файлу cookie **`/run/tor/control.authcookie`** (hex в команде `AUTHENTICATE`). Пользователь **php-fpm** (часто `www-data`) должен входить в группу **`debian-tor`**, иначе будет «Connection refused» / нет доступа к cookie:

```bash
sudo bash install/scripts/tor_php_debian_tor.sh
```

Опционально можно добавить **TCP ControlPort** (`127.0.0.1:9051`) скриптом **`install/scripts/tor_append_control_port.sh`** — тогда сработает и подключение по `control_host` / `control_port` из `config.json`. Поле **`control_password`**: непустое — пароль; пустое — cookie (путь переопределяется **`control_cookie_path`** в `diagnostics.telegram.tor`).

Если включён **`newnym_before_send`**, перед каждой отправкой алерта тоже вызывается NEWNYM — без доступа к control (сокет или TCP) это не сработает.

## 5. Service after reboot

```bash
systemctl is-active tor@default
```

Should print `active`.

## 6. Nginx / «Ждать чат»

Один запрос `GET .../diagnostics/telegramWait` на сервере выполняет **deleteWebhook** и **getUpdates** подряд. Если суммарное время curl больше, чем **`fastcgi_read_timeout`** у nginx (часто **60 s**), в логе будет `upstream timed out while reading response header`, браузер не получит JSON, а в интерфейсе может показаться истечение ожидания `/start`.

В конфиге `server` / `location` для PHP (frontend) при необходимости увеличьте, например:

```nginx
fastcgi_read_timeout 120s;
```

После правки: `sudo nginx -t && sudo systemctl reload nginx`.

Готовый фрагмент (подстройте `SCRIPT_FILENAME` под ваш vhost): **`install/nginx/rbt-diagnostics-telegram-timeout.conf.example`**. В коде для `telegramWait` включён режим коротких запросов к Telegram (`_telegram_wait_ui_fast`), чтобы укладываться в типичный `fastcgi_read_timeout`; при HTTP **504** в UI см. этот файл.

## 7. Только Tor, без своих прокси

Обход блокировки строится на **локальном Tor**: SOCKS `127.0.0.1:9050`, при необходимости **мосты** (раздел 2 и скрипт **`install/scripts/tor_enable_snowflake.sh`**) и **доступ к Tor Control** для NEWNYM (раздел 4: Unix-сокет или TCP). Прямой доступ к `api.telegram.org` с сервера при жёсткой блокировке обычно бесполезен — выставьте **`tor.fallback_direct`: `false`**, чтобы не тратить время на заведомо мёртвый маршрут.

В коде RBT для curl используется **`CURLPROXY_SOCKS5_HOSTNAME`**: имя `api.telegram.org` уходит в Tor и резолвится **там** (аналог **`ProxyConnector` + `rdns=True`** в Python). Раньше для всех запросов включался принудительный IPv4 на стороне PHP — при блокировке DNS к Telegram это мешало; для трафика через SOCKS это отключено.

## 8. PHP-FPM после `tor.service` (как `After=tor` у airy)

Чтобы php-fpm не поднимался раньше Tor, добавьте drop-in к юниту php-fpm (имя зависит от версии PHP), см. **`install/systemd/php-fpm.service.d.tor.conf.example`**.

## 9. Идемпотентно: TCP ControlPort в torrc (опционально)

Скрипт **`install/scripts/tor_append_control_port.sh`** (запуск с `sudo`) добавляет строки `ControlPort 127.0.0.1:9051` и `CookieAuthentication 1`, если их ещё нет. На многих системах уже достаточно **Unix-сокета** `/run/tor/control` (раздел 4); TCP нужен, если хотите управлять Tor только по `127.0.0.1:9051`. Затем перезапустите Tor.

Дополнительно: **`install/scripts/tor_php_debian_tor.sh`** — добавляет `www-data` в группу `debian-tor` и перезапускает php-fpm (нужно для cookie и сокета). **`install/scripts/tor_enable_snowflake.sh`** — пакет `snowflake-client`, фрагмент `UseBridges` + Snowflake в `/etc/tor/torrc.d/`, перезапуск Tor.

## 10. Чеклист «как в Airy»

| Шаг | Действие |
|-----|----------|
| 1 | `tor` запущен, `9050` слушает, `curl --socks5-hostname …` до check.torproject.org и до `api.telegram.org` проходит |
| 2 | При цензуре — мосты (Snowflake / obfs4) в `torrc` |
| 3 | Доступ к Tor Control для NEWNYM: Unix-сокет + группа `debian-tor` для php-fpm (раздел 4), при желании TCP ControlPort (раздел 9) |
| 4 | В RBT: весь Bot API через SOCKS `127.0.0.1:9050`, резолв имён внутри Tor (см. §7) |
| 5 | **Не** полагаться на «httpx + uvloop + SOCKS» в PHP — здесь **curl** в php-fpm |
| 6 | php-fpm стартует после `tor` (§8), nginx **`fastcgi_read_timeout 120s`** для «Ждать чат» (§6) |

## 11. Автопроверка и ротация мостов (Snowflake)

Автоматически **скачать** новые мосты с BridgeDB без капчи нельзя — пул хранится локально в **`/etc/tor/rbt-bridge-pool.txt`** (строки `Bridge snowflake …`, по одной на мост). Добавляйте мосты вручную с [bridges.torproject.org](https://bridges.torproject.org/) (тип snowflake). Для **ротации** нужно **не меньше двух** разных строк в пуле.

Скрипты в `install/scripts/`:

| Скрипт | Назначение |
|--------|------------|
| `tor_bridge_pool.example.txt` | образец пула |
| `tor_bridge_pool_apply.sh` | пересобрать `/etc/tor/torrc.d/50-rbt-snowflake.conf` из пула |
| `tor_bridge_pool_rotate.sh` | сдвинуть первый мост в конец пула, применить, перезапустить Tor |
| `tor_bridge_healthcheck.sh` | проверка `curl` через SOCKS (код 0/1) |
| `tor_bridge_monitor.sh` | при **двух подряд** сбоях проверки — ротация и перезапуск Tor |
| `tor_bridge_monitor_install.sh` | установка **systemd timer** (раз в ~15 мин после загрузки) |

Установка таймера и состояния (`/var/lib/rbt-tor/`):

```bash
sudo bash /opt/rbt/install/scripts/tor_bridge_monitor_install.sh
systemctl status tor-bridge-monitor.timer
journalctl -u tor-bridge-monitor.service -n 30 --no-pager
```

Переменные окружения (для `service` можно задать через `Environment=` в drop-in): `RBT_TOR_HEALTHCHECK_URL` (по умолчанию check.torproject.org), `RBT_TOR_HEALTHCHECK_TIMEOUT`, `RBT_TOR_HEALTHCHECK_MAX_FAILS` (сколько неудачных проверок подряд до ротации, по умолчанию 2), `RBT_TOR_POOL`.
