# Виртуальная QR-панель: открытие по DTMF из мобильного приложения

Цель: при звонке с панели **`model = virtual.json`** жилец в официальном приложении нажимает «Открыть» — приложение шлёт **DTMF** (часто **SIP INFO**). Asterisk поднимает AMI-события; слушатель **`vdom_ami_dtmf_listener.php`** сверяет цифры с настройкой панели **`dtmf`**, дергает **`ext.doorOpeningUrls[0]`**.

Этот документ — пошаговое развёртывание на **новой** машине с уже установленным стеком из [10.asterisk.md](10.asterisk.md) и [07.install.md](07.install.md).

## 1. Предпосылки

- Asterisk с **PJSIP**, модуль **`res_pjsip_dtmf_info`** (приём SIP INFO как DTMF — типичная сборка 20+).
- Симлинк конфигов: `/etc/asterisk` → каталог репозитория с **`extensions.lua`**, **`manager.conf`**.
- Redis из `server/config/config.json` — тот же инстанс, что использует Lua dialplan (`asterisk/config.lua` → redis).
- В БД домофон: **`houses_domophones.model = virtual.json`**, в **`ext`** — **`doorOpeningUrls`** (массив URL) и **`dtmf`** на самой панели.

## 2. Конфиг приложения (PHP)

В **`server/config/config.json`** (не коммитится; образец — **`server/config/config.sample.json5`**) добавьте блок **`vdom_dtmf`**:

```json
"vdom_dtmf": {
    "ami": {
        "host": "127.0.0.1",
        "port": 5038,
        "username": "rbtdom",
        "secret": "СЛУЧАЙНЫЙ_ДЛИННЫЙ_СЕКРЕТ"
    }
}
```

Поле **`open_api_key`** в коде слушателя не используется (зарезервировано); можно опустить.

## 3. AMI в Asterisk

В **`asterisk/manager.conf`** (в репозитории; на сервере обычно тот же файл через symlink) должен быть пользователь **`[rbtdom]`**:

- **`secret`** — **тот же**, что `vdom_dtmf.ami.secret` в `config.json`.
- **`deny` / `permit`** — по умолчанию только **127.0.0.1**; если слушатель на другом хосте — расширьте `permit`.
- **`read`** минимум: `system,call,log,dtmf`.

После правок:

```bash
sudo asterisk -rx "manager reload"
```

## 4. Код (уже в репозитории)

| Компонент | Назначение |
|-----------|------------|
| **`asterisk/extensions.lua`** → `mobileIntercom` | Для **`domophoneData.model == "virtual.json"`** в Redis **`vdom_dtmf_ctx:<linkedid>`** на 600 с: `domophoneId`, `flatId`, `flatNumber`, `dtmf`, **`doorUrl`** (первый URL из `doorOpeningUrls`). |
| **`server/asterisk.php`** | Для мобильного SIP (`2…` **10** цифр) **`dtmf_mode` => `auto`** — чтобы принимались и RFC2833, и SIP INFO. |
| **`server/cli/vdom_ami_dtmf_listener.php`** | Подключение к AMI, разбор **`DTMF` / `DTMFEnd`**, накопление строки до совпадения с `dtmf`, **GET** по `doorUrl`, логи. |
| **`server/hw/ip/domophone/models/virtual.json`** | Модель виртуальной QR-панели в админке. |

После правок Lua:

```bash
sudo asterisk -rx "module reload pbx_lua"
```

## 5. Служба systemd (один процесс)

Не запускайте слушатель вручную несколько раз — будут дубли в логах и повторные открытия.

**Почему в логе раньше могли быть `startup` / `ami_login_ok` каждую минуту:** строка `startup` пишется только при **старте нового процесса** PHP (не при переподключении к AMI внутри одного процесса). Значит, параллельно с systemd кто-то ещё запускал тот же скрипт (частый вариант — лишняя строка в **cron** вроде `* * * * * php …/cli/vdom_ami_dtmf_listener.php`) или было несколько unit’ов. Второй процесс теперь **сразу выходит** с сообщением в stderr (flock на `server/logs/.vdom_ami_dtmf_listener.lock`), не дописывая мегабайты в `vdom_ami_listener.log`.

**Шум в `vdom_ami_listener.log`:** по умолчанию (`vdom_dtmf.ami.verbose_log` не задан или `false`) в файл попадают только ошибки и важные события (например `dtmf_open_done`); подробный поток DTMF и каждый логин AMI — только при **`"verbose_log": true`** в `vdom_dtmf.ami`.

```bash
sudo ln -sf /opt/rbt/install/systemd/rbt-vdom-ami-dtmf.service /etc/systemd/system/rbt-vdom-ami-dtmf.service
sudo systemctl daemon-reload
sudo systemctl enable --now rbt-vdom-ami-dtmf
```

Пользователь **`rbt`** должен иметь право читать `config.json` и писать в **`/opt/rbt/server/logs/`** (каталог создаётся скриптом при необходимости).

## 6. Логи (проверка)

| Файл | Содержимое |
|------|------------|
| **`server/logs/vdom_ami_listener.log`** | По умолчанию: ошибки AMI и **`dtmf_open_done`**. При **`vdom_dtmf.ami.verbose_log`: true** — также старт, **`ami_login_ok`**, каждое **`dtmf_event`**, переподключения. |
| **`server/logs/vdom_dtmf_door.log`** | Успешное/неуспешное открытие строкой вида `…\tdtmf\t<domophoneId>\t<flatId>\t<flatNumber>\tdoor0\tok\|fail`. |
| **`server/logs/vdom_ami_dtmf.stderr.log`** | Stderr unit (PHP-ошибки). |

## 7. Поведение при нескольких панелях

У каждого звонка свой **`linkedid`** → свой ключ Redis. Новые панели с **`virtual.json`** и своими **`doorOpeningUrls` / `dtmf`** работают так же.

Запасной режим в слушателе: если по AMI не найден ключ, но в Redis **ровно один** `vdom_dtmf_ctx:*` — он используется (удобно для отладки; при **одновременных** двух звонках на две виртуальные панели не полагайтесь на этот fallback).

## 8. Частые сбои

- **Нет `ami_login_ok`** — неверный секрет, нет секции `[rbtdom]`, не сделан `manager reload`.
- **Есть `dtmf_event`, нет открытия** — пустой **`doorUrl`** в контексте (нет `doorOpeningUrls` в `ext` или Lua не смог разобрать таблицу).
- **Нет DTMF в AMI** — проверить `dtmf_mode` мобильного endpoint, модуль `res_pjsip_dtmf_info`, `pjsip set logger on` на время теста.

## 9. Ссылки в коде приложения (Android)

В разговоре «Открыть» шлёт только DTMF (SIP INFO у Linphone), не HTTP `openDoor`. См. репозиторий **`SmartYard-Android`**: `LinphoneProvider.sendDtmf`, `IncomingCallActivity.openDoor`.
